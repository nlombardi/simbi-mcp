## Key observations

### Page coordinate system
- Page dimensions: **1280 ├Ś 720** (not 800 ŌĆö correct this in Phase 3 plan)
- Visual `position.x/y` are in points that match pixel coords at 100% zoom
- `z` values are 0, 1000, 2000 (tab order, not depth)
- `tabOrder` matches `z` value

### What SimBI needs to generate (Report side)

Must create these files (minimum for a valid report):
1. `definition.pbir` ŌĆö static content, just the path to SemanticModel
2. `definition/version.json` ŌĆö always `{"version": "2.0.0"}`
3. `definition/report.json` ŌĆö theme ref + settings (can use template)
4. `definition/pages/pages.json` ŌĆö page order list
5. `definition/pages/<guid>/page.json` ŌĆö one per page
6. `definition/pages/<guid>/visuals/<guid>/visual.json` ŌĆö one per visual
7. `StaticResources/SharedResources/BaseThemes/CY25SU10.json` ŌĆö copy from ref

Do NOT generate: `.pbi/cache.abf` (binary, auto-generated by Desktop on open)

### What SimBI needs to generate (SemanticModel side)

The MS Power BI MCP generates the SemanticModel folder. SimBI orchestrates it via
`measure_operations` and `table_operations` ŌĆö we do not write TMDL by hand.

## Visual JSON structure

### Schema URL (all visuals)
```
https://developer.microsoft.com/json-schemas/fabric/item/report/definition/visualContainer/2.3.0/schema.json
```

### Card visual
```json
{
  "$schema": "...",
  "name": "<guid>",
  "position": { "x": 0, "y": 0, "z": 1000, "height": 202.23, "width": 280.74, "tabOrder": 1000 },
  "visual": {
    "visualType": "card",
    "query": {
      "queryState": {
        "Values": {
          "projections": [
            {
              "field": { "Measure": { "Expression": { "SourceRef": { "Entity": "Sales" } }, "Property": "Total Revenue" } },
              "queryRef": "Sales.Total Revenue",
              "nativeQueryRef": "Total Revenue"
            }
          ]
        }
      }
    },
    "drillFilterOtherVisuals": true
  }
}
```

Note: use `Measure` reference (not `Aggregation`) when binding to a SimBI-created named measure.

## Field reference patterns

| Scenario | Pattern | When to use |
|---|---|---|
| Named DAX measure | `{"Measure": {"Expression": {"SourceRef": {"Entity": "T"}}, "Property": "Measure Name"}}` | SimBI-generated measures (Phase 1 output) |
| Column with implicit SUM | `{"Aggregation": {"Expression": {"Column": {...}}, "Function": 0}}` | Raw column fallback (avoid in SimBI) |
| Dimension column | `{"Column": {"Expression": {"SourceRef": {"Entity": "T"}}, "Property": "ColumnName"}}` | Axis, slicer field, series |

**SimBI should always use `Measure` references** for values, never raw `Aggregation`. Named measures = the semantic layer is authoritative.

## `queryRef` and `nativeQueryRef` conventions

- `queryRef`: `"<Entity>.<Property>"` e.g. `"Sales.Total Revenue"`
- `nativeQueryRef`: just the property name e.g. `"Total Revenue"`
- Both are informational/display ŌĆö Power BI derives them from the field binding

## report.json (static template ŌĆö copy as-is)

The `themeCollection`, `objects`, `resourcePackages`, and `settings` blocks can be
copied verbatim from this reference for all generated reports. Only the theme name
matters and CY25SU10 (shipped with Power BI Desktop) is always available.

## TMDL observation (SemanticModel)

The MS MCP annotates models it created:
```
annotation PBI_ProTooling = ["MCP-PBIModeling","DevMode"]
```

This confirms the MCP successfully authored the semantic model. The TMDL format is
human-readable and could be hand-edited if needed, but SimBI delegates all model
authoring to the MS MCP tools.

## Round-trip test result

Confirmed: file opens in Power BI Desktop, visuals render correctly. Moving
a visual 100px right changes only `position.x` in visual.json ŌĆö no other fields.

4. Must include `StaticResources/SharedResources/BaseThemes/CY25SU10.json` (copy from ref)
5. `definition.pbir` path is relative (`../ReferenceDashboard.pbib.SemanticModel`) ŌĆö keep this pattern
